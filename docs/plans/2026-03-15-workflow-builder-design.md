# Workflow Builder Tools Design

**Date:** 2026-03-15
**Goal:** Add `jobtread_create_workflow` and `jobtread_update_workflow` MCP tools that let AI guide users through building JobTread automation workflows.

---

## Context

JobTread has a native workflow automation system with 66 trigger types, 41 action types, and a 10-workflow-per-org limit. Workflows are trigger-based automations with nested action trees that use `{{trigger.field}}` template syntax.

**Existing read tools:** `list_workflows`, `get_workflow`, `list_workflow_runs` — already deployed.

**Key constraint:** Orgs can only have 10 workflows, so they are trigger-dependent. The AI should always check existing workflows first and modify them (add action branches) rather than creating duplicates with the same trigger.

---

## Approach: Two-Tool Model

Two tools matching existing create/update conventions: `jobtread_create_workflow` + `jobtread_update_workflow`.

### AI Behavioral Flow

1. User describes desired automation
2. AI calls `list_workflows` to check for existing trigger matches
3. If a workflow with the right trigger exists → `get_workflow` → merge new actions → `update_workflow`
4. If no match → `create_workflow` with `isActive: false`
5. AI explains what was built, suggests user review in JobTread UI before activating

---

## Tool Schemas

### `jobtread_create_workflow`

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| name | string | yes | Workflow name |
| triggerTypeId | string | yes | One of 66 trigger types (e.g., `taskCreated`, `jobUpdated`, `scheduled`) |
| triggerInput | object | no | Trigger configuration (e.g., cron for scheduled triggers) |
| actions | array | yes | Nested action tree |
| isActive | boolean | no | Default: `false` — never auto-activate |

**Description:**
> Create a new automation workflow. BEFORE creating, ALWAYS call list_workflows first to check if a workflow with the same trigger type already exists — if so, use update_workflow to add actions instead. Orgs are limited to 10 workflows. Start with isActive: false so the user can review before activating.
>
> Common patterns: filter → action (gate by condition), findMembership → assignAccess (team routing), trigger templates like {{trigger.job.id}}, {{trigger.task.taskType.name}}, {{trigger.job.custom:FIELD_ID}}.

### `jobtread_update_workflow`

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| workflowId | string | yes | Workflow ID |
| name | string | no | New name |
| triggerTypeId | string | no | Change trigger type |
| triggerInput | object | no | Change trigger config |
| actions | array | no | REPLACES entire action tree |
| isActive | boolean | no | Activate/deactivate |

**Description:**
> Update an existing workflow's name, trigger, active status, or action tree. The actions array REPLACES the entire action tree — fetch the workflow first with get_workflow, merge your changes, then send the complete tree. Use this to add new action branches to existing workflows that share the same trigger.

## Action Tree Format

Each action in the array:

```json
{
  "typeId": "filter",
  "name": "Optional human label",
  "input": { "filter": { "expression": { "=": [...] } } },
  "actions": [
    {
      "typeId": "createComment",
      "input": { "targetId": "{{trigger.job.id}}", "message": "Hello" },
      "actions": []
    }
  ]
}
```

Action IDs are auto-generated by Pave — AI does NOT provide them. Child actions reference parent results via `{{parentActionId.field}}`.

### Action Types Reference (41 total)

| Category | Types |
|----------|-------|
| **Filter/Logic** | `filter`, `delay`, `loop`, `transform`, `formatDateTime` |
| **Find (13)** | `findJob`, `findAccount`, `findContact`, `findMembership`, `findTask`, `findDocument`, `findCostItem`, `findCostGroup`, `findDailyLog`, `findTimeEntry`, `findEvent`, `findLocation`, `findFile` |
| **Create (9)** | `createJob`, `createTask`, `createComment`, `createDailyLog`, `createDocument`, `createCostItem`, `createCostGroup`, `createTimeEntry`, `createNotification` |
| **Update (11)** | `updateJob`, `updateTask`, `updateAccount`, `updateContact`, `updateDocument`, `updateCostItem`, `updateCostGroup`, `updateLocation`, `updateDailyLog`, `updateTimeEntry`, `updateMembership` |
| **Other (3)** | `assignAccess`, `query`, `webhook` |

### Trigger Types Reference (66 total)

**Event triggers (60):** 4 per entity (created, updated, deleted, restored) × 15 entities: job, task, account, contact, location, document, costItem, costGroup, costCode, dailyLog, timeEntry, comment, file, event, membership

**Reminder triggers (5):** taskReminder, documentReminder, jobReminder, eventReminder, paymentReminder

**General triggers (2):** scheduled, webhook

## Common Workflow Patterns

### Pattern 1: Notification on Task Creation
```
Trigger: taskCreated
→ filter (task type = "Site Visit")
  → createComment (customer-visible reminder message on job)
```

### Pattern 2: Team Assignment by Project Type
```
Trigger: jobUpdated
→ filter (custom field "Project Type" = "Barndominium")
  → findMembership (by email)
    → assignAccess (to job)
→ filter (custom field "Project Type" = "Post Frame")
  → findMembership (by email)
    → assignAccess (to job)
```

### Pattern 3: Pipeline Advancement
```
Trigger: documentApproved (customerOrder)
→ updateJob (set status custom field to "Committed")
→ createTask (add "Kickoff Meeting" to-do)
```

### Pattern 4: Daily Log Reminder
```
Trigger: timeEntryCreated
→ delay (8 hours)
  → filter (no daily log exists for today)
    → createNotification (remind to fill in daily log)
```

## Pave Mutations

### createWorkflow
```javascript
const data = await ctx.pave({
  createWorkflow: {
    $: {
      name,
      triggerTypeId,
      triggerInput: triggerInput || {},
      actions,
      isActive: isActive ?? false,
    },
    createdWorkflow: {
      id: {}, name: {}, isActive: {}, triggerTypeId: {}, url: {}, actions: {},
    },
  },
});
```

### updateWorkflow
```javascript
const updates = {};
if (name !== undefined) updates.name = name;
if (triggerTypeId !== undefined) updates.triggerTypeId = triggerTypeId;
if (triggerInput !== undefined) updates.triggerInput = triggerInput;
if (actions !== undefined) updates.actions = actions;
if (isActive !== undefined) updates.isActive = isActive;

const data = await ctx.pave({
  updateWorkflow: {
    $: { id: workflowId, ...updates },
    updatedWorkflow: {
      id: {}, name: {}, isActive: {}, triggerTypeId: {}, actions: {},
    },
  },
});
```

## Safety Guardrails

1. **Default `isActive: false`** — workflows are never auto-activated
2. **10-workflow limit warning** — if org is at 8+, warn user in response
3. **Check-before-create** — tool description instructs AI to always call `list_workflows` first
4. **No delete** — consistent with MCP no-delete policy
5. **Actions replace entire tree** — on update, AI must fetch existing workflow first and merge
